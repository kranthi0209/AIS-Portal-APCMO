🖥️ e-Office Performance — Scoring Methodology

How each AIS officer's file-handling performance is converted into a fair, absolute 0–100 score — four indicators, each scored on its own fixed scale, blended by weight, then scaled by a workload (difficulty) factor.

1 · Purpose & design principles

The e-Office system records how officers move government files — how many they receive, clear, and how fast. This score measures file-handling efficiency on one 0–100 scale that is absolute: an officer's number reflects their own performance against fixed benchmarks, and does not change because of who else is on screen.

PrincipleHow we achieve it
Absolute, not relativeEach indicator is scored on a fixed 0–100 scale (the same yardstick for everyone, every period). Filtering the list changes who is shown and the rank order — never an officer's score.
Transparent & auditableEvery sub-score has an explicit formula and benchmark; an officer can recompute their own score by hand.
Weights mean what they sayA weighted average of 0–100 sub-scores → a 45% indicator really is 45% of the result.
Robust to outliersSpeed is median-led with small-sample shrinkage; consistency uses the average–median gap; volume is log-scaled. No single odd file or mega-inbox distorts the scale.
Bigger workloads get creditA log-scaled workload (difficulty) factor (0.50–1.00) rewards officers handling more files — clearing 8,000 well beats clearing 50 well.

2 · The raw data & the four indicators

For each officer the e-Office system supplies: Opening Balance, Files Received, Files Processed, Files Pending, Average and Median processing time (minutes), and Days Active. From these we derive four indicators (each mapped to 0–100) plus the workload factor:

1Clearance RateFiles Processed ÷ (Opening + Received) × 10025%
2Low Pendency(1 − Files Pending ÷ (Opening + Received)) × 10010%
3Disposal Speed100 × 1200 ÷ (1200 + T),  T = robust, shrunk time/file45%
4Consistencyexp(−(Avg − Median) ÷ (Median + 1)) × 10020%

What changed & why. The earlier model used six indicators scored as percentiles against the on-screen cohort (so scores shifted when you filtered), with separate median- and average-speed metrics and a standalone "Days Active" metric. The current model: (a) merges median + average into one robust Disposal Speed with small-sample shrinkage; (b) folds Days Active into the workload factor; and (c) scores every indicator on a fixed absolute scale instead of a peer percentile — so an officer's score is stable and comparable across periods.

3 · How posts are handled, and tenure

Unlike the Perception, KPI and GoI-Funds scores, the e-Office score is inherently officer-level. The e-Office system logs file movement against the officer regardless of whether the post is Secretariat, HoD or District — a file is a file — so there is no per-post split for this score.

Tenure across multiple or overlapping postings is captured by Days Active: the unique calendar days the officer was engaged, merging overlapping post periods so a day on which two posts overlap is never double-counted. Days Active now feeds the workload factor (§5) rather than being a standalone indicator.

Officers with no file movement in the window are shown separately with a score of 0 and ranked last.

4 · The scoring method, step by step

Step 1 — Disposal Speed: a robust, shrunk time per file

We only have each officer's average, median and count of processing times. The median is robust (a few monster files can't distort it); the average is sensitive to a slow tail. We blend them median-first, then shrink toward a fixed benchmark so an estimate from very few files isn't taken at face value (empirical-Bayes / James–Stein):

T_eff = 0.65 · Median + 0.35 · Average
T* = ( n · T_eff + k · REF ) ÷ ( n + k )   n = Files Processed,  k = 30,  REF = 1200 min

With many files, T* ≈ T_eff; with few files, T* is pulled toward the 1,200-minute benchmark, so a lucky handful of fast files cannot fake "fast."

Step 2 — Score each indicator 0–100 (absolute)

Clearance = min(100, Processed ÷ (Opening + Received) × 100)
LowPendency = (1 − Pending ÷ (Opening + Received)) × 100
Speed = 100 × REF ÷ (REF + T*)  (a file at REF = 1200 min scores 50)
Consistency = exp( −(Average − Median) ÷ (Median + 1) ) × 100

No comparison to other officers is involved — these are fixed transforms of the officer's own numbers.

Step 3 — Efficiency = weighted average of the four sub-scores

Efficiency = 0.25·Clearance + 0.10·LowPendency + 0.45·Speed + 0.20·Consistency

Because the weights sum to 1 and every sub-score is 0–100, Efficiency is automatically 0–100 and each indicator contributes exactly its stated weight.

Step 4 — Apply the workload (difficulty) factor → final score

e-Office Score = Efficiency × Workload factor  (capped 0–100)

Step 5 — Rank

Officers are ordered by the final score, highest first. (The rank is the only relative part — it just sorts the absolute scores.) The four sub-scores and the workload factor are shown in the score-cell tooltip.

5 · The workload (difficulty) factor

Clearing a tiny inbox well is easier than clearing a huge one well. So Efficiency is scaled by a factor built from the officer's workload — total files faced and daily intensity (files per active day) — on a log scale against fixed targets (no cohort min–max, so it too is absolute):

P = (Opening + Received) ÷ max(Days Active, 1)  (files per active day)
L = ln(1 + Opening + Received) + 0.5 · ln(1 + P)
L_full = ln(1 + 5000) + 0.5 · ln(1 + 60)  (full-credit target)
Workload factor = 0.50 + ( 1 − 0.50 ) × min(1, L ÷ L_full)  ∈ [0.50, 1.00]
  • An officer at or above the target (≈ 5,000 files at ~60/day) gets the full ×1.00.
  • The lightest workload keeps ×0.50 — a 50% discount reflecting the lower difficulty.
  • The logarithm compresses the huge spread of file counts so one mega-inbox doesn't dominate; the 0.50 floor ensures a genuinely excellent low-volume officer still scores respectably.

6 · Glossary of statistical terms

Absolute (fixed-scale) score
A 0–100 number computed from the officer's own data against fixed benchmarks — independent of the peer set, so filtering never changes it.
Robust effective time (T_eff)
0.65·Median + 0.35·Average — median-led (resistant to a few very slow files) with a touch of the average to catch a slow tail.
Empirical-Bayes / shrinkage (T*)
Pulling a small-sample estimate toward a benchmark in proportion to how few observations it rests on, so a handful of files can't produce an extreme, unreliable speed.
Benchmark / reference time (REF)
The fixed time per file (1,200 min) that scores 50/100 on Speed; faster scores higher, slower lower.
Weighted arithmetic mean
An average in which each indicator counts in proportion to its weight; the weights sum to 1.
Workload (difficulty) factor
A 0.50–1.00 multiplier from the log of (Opening + Received) and daily intensity (files per active day) against fixed targets; rewards a larger, more intense workload.
Logarithm (ln)
Compresses very large counts so a few mega-volume officers don't dominate the workload factor.
Relative skew (consistency)
(Average − Median) ÷ (Median + 1): how far the average sits above the median — a proxy for a long tail of slow files. Smaller = steadier.
Unique days worked
Distinct calendar days of engagement, with overlapping post periods merged so no day is double-counted.
